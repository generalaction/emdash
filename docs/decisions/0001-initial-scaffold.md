# ADR-0001: emdash-dev initial scaffold (Tauri 2 + Rust)

## Status

Accepted

## Context

EMD-5 introduces **emdash-dev**, a Tauri 2 + Rust rewrite of emdash that
ships as a separate product alongside the existing Electron app. This first
PR is the discipline-establishing scaffold — module organisation, version
pins, and verification mechanisms set here propagate to every later issue
in the rewrite track.

Several non-obvious decisions emerged during the scaffold that future
readers would otherwise have to reverse-engineer from `Cargo.toml`, build
output, or stale upstream docs. This ADR records them.

## Decision

### 1. Version pins

| Crate              | Pin             | Why                                                                                                                                       |
| ------------------ | --------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `tauri`            | `=2.11.1`       | Latest stable. Earlier 2.x.y in the same minor (e.g. 2.9.5) hit a `wry` Send+Sync regression in `tauri-runtime-wry::with_new_window_req_handler` that breaks compilation. |
| `tauri-build`      | `=2.6.1`        | Released on its own cadence — current latest. Does **not** track `tauri`'s minor; expect them to diverge.                                 |
| `tauri-specta`     | `=2.0.0-rc.25`  | v2 has been in RC for a while; rc.25 is current. We accept RC instability in exchange for typed bindings; a stable release will trigger an ADR update. |
| `specta`           | `=2.0.0-rc.25`  | Lockstep with `tauri-specta`.                                                                                                             |
| `specta-typescript`| `=0.0.12`       | Pre-1.0 separate crate for the TS exporter. Pinned exactly because every patch has been API-breaking.                                     |

All four are pinned **exactly** (`=`) so CI and developer machines resolve to
the same versions despite the volatile pre-release ecosystem.

### 2. Crate layout — single crate, workspace deferred

```
src-tauri/
  Cargo.toml                # one package: emdash-dev
  build.rs                  # tauri_build::build() + capability-allowlist check
  tauri.conf.json
  rust-toolchain.toml       # at repo root, channel = stable
  src/
    lib.rs                  # pub mod greeting; pub mod shell_env;
    main.rs                 # the Tauri app binary entry point
    app.rs                  # tauri::Builder + tauri-specta wiring (bin-tree only)
    shell_env.rs            # AppHandle-free; ported from Electron emdash userEnv.ts
    greeting.rs             # pure domain implementation behind the greet command
    commands/               # #[tauri::command] glue; AppHandle-free for now
      mod.rs
      greet.rs
      path.rs
    bin/
      emdash-cli.rs         # near-empty CLI — exists to enforce that domain code
                            # stays Tauri-independent
  ui/                       # vendored Vite + React app (pnpm workspace member)
    src/
      App.tsx
      bindings.ts           # auto-generated by tauri-specta, committed
```

**Why single crate now, not a Cargo workspace.** A workspace split — one crate
for pure domain modules, one for the Tauri app — would *linker-enforce* the
Tauri-independence invariant. But it adds setup cost (workspace `Cargo.toml`,
per-crate `Cargo.toml`s, inter-crate version management) that isn't justified
until mobile (iOS/Android) lands. The current discipline is source-level:
`bin/emdash-cli.rs` links the same `emdash_dev` library modules as the app, and
`tests/domain_boundaries.rs` fails if modules exported from `lib.rs` import
Tauri command/runtime APIs.

**Why `#[tauri::command]` functions live outside `lib.rs`'s exports.** Command
functions are binary glue: they translate Tauri invoke calls into domain module
calls. The domain implementation (`emdash_dev::greeting::greet`,
`emdash_dev::shell_env::*`) remains reachable from integration tests and the CLI
without carrying webview runtime annotations into the library surface.

### 3. shell_env source — ported from Electron emdash, not Helmor

`src-tauri/src/shell_env.rs` is a faithful Rust translation of
`src/main/utils/userEnv.ts` (lines 110-159) plus `src/main/utils/childProcessEnv.ts`.

This deviates from EMD-5's literal text ("ported from Helmor"). The reasoning:

- The Electron version invokes `$SHELL -ilc 'env'` (login **plus** interactive).
  Helmor's reference uses `$SHELL -l -c env` (login only). Interactive
  invocation matters because some shell init paths (oh-my-zsh, mise, starship)
  only execute under `-i`.
- The Electron version has been hardened against AppImage runtime leakage
  (issue #1679) — entries under `$APPDIR` or `/tmp/.mount_*` are scrubbed
  before the probe shell sees them. Without this, login-shell hooks that
  resolve a binary by name through `$PATH` (mise, starship, oh-my-zsh) can
  re-enter the AppImage and fork-bomb the app on Linux.
- The `PRESERVE_KEYS` blocklist (AppImage runtime vars, `NODE_ENV`) is
  load-bearing. Reproducing it from the Helmor version would require
  re-discovering each gotcha.

The Tauri version intentionally **captures** into a `OnceLock<ShellEnv>` rather
than requiring every caller to run the shell probe. At startup the app calls
`shell_env::apply_login_shell_env_to_process()` once, which applies the captured
map to `std::env` so future shell-outs inherit the Finder/Spotlight-safe PATH.
The captured map is also exposed via `shell_env::external_tool_env()` for future
spawn helpers that should pass env explicitly.

### 4. Capability allowlist — separate from Tauri's permission system

Tauri 2's `capabilities/*.json` files configure the built-in permission system
(filesystem, shell, plugin permissions for specific windows). They do **not**
gate custom `#[tauri::command]` functions — those are exposed as soon as they
appear in `invoke_handler`.

EMD-5 requires "any new command without an entry [to] fail the build". Our
implementation:

- `src-tauri/allowed-commands.json` (at the crate root, **not** in
  `capabilities/`, because Tauri parses every JSON in that directory as one of
  its own capability files and rejects unknown schemas).
- `build.rs` parses the channel names emitted into `ui/src/bindings.ts` and
  cross-references against `allowed-commands.json`. Drift in either direction
  panics the build. `cargo:rerun-if-changed` triggers re-check on either file.
- Two-direction enforcement: a command in bindings.ts but not in the allowlist
  fails (forgot to allow), and a command in the allowlist but not in bindings
  also fails (orphan entry, likely a removed command).

### 5. bindings.ts is committed and verified

The generated TypeScript bindings live in `src-tauri/ui/src/bindings.ts` and
are **committed** rather than `.gitignore`'d. Two consumers verify it stays in
sync with the Rust command set:

- `build.rs` (above): drift between bindings and allowlist fails the build.
- `tests/wire_format.rs`: an `insta` snapshot of bindings.ts content. Reviews
  are forced when a command's signature changes (parameter renamed, type
  swapped, return value modified).

CI regenerates bindings via `cargo run --bin emdash-dev -- --export-bindings`
(a non-windowed code path on the main binary), then asserts
`git diff --exit-code src-tauri/ui/src/bindings.ts`.

### 6. Placeholder app icon

`src-tauri/icons/icon.png` is a 256×256 indigo PNG generated by a one-line
Python script. It exists only to satisfy `tauri::generate_context!`, which
requires *some* icon at compile time even with `bundle.active: false`. Real
icons (`.icns`, `.ico`, multi-resolution PNGs) are a follow-up before the
first dev-channel release; they will not be hand-authored — `cargo tauri icon`
generates the set from a single source PNG.

### 7. Bundling disabled in this PR

`tauri.conf.json` sets `bundle.active: false`. CI uses `cargo build`
(unsigned binaries) on every PR; bundling (DMG, MSI, AppImage) is deferred
until the icon set lands and signing keys are provisioned.

## Consequences

**Easier:**

- Future emdash-dev PRs inherit the version pins, layout, and verification
  mechanisms without re-litigating them.
- The Tauri-independence invariant is a grep check, not a workspace split —
  cheap to enforce.
- Snapshot tests catch wire-format regressions before they reach the renderer.

**Harder:**

- Bumping `tauri`, `tauri-build`, `tauri-specta`, or `specta-typescript` is now
  a deliberate 4-line PR rather than a transitive `cargo update`. We accept
  this in exchange for reproducibility against an RC ecosystem.
- A future workspace split (when mobile lands) will require moving the current
  library modules into a dedicated domain crate. That migration is bounded by
  the small `lib.rs` export surface.

**Triggers to revisit:**

- `tauri-specta` reaches a stable 2.0.0 release → bump and remove the RC note.
- Mobile (iOS/Android) work begins → split into workspace; promote `lib.rs`
  modules into a `crates/emdash-domain` crate so `bin/emdash-cli` link-enforces
  the invariant instead of relying on grep.
- A new command needs `tauri::AppHandle` access (e.g., to open a window or
  emit an event) → that command stays in the bin tree under `src/app/`, not
  in the domain modules exported by `lib.rs`.
